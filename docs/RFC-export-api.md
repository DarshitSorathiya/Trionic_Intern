# 1. Purpose

This RFC proposes the architecture for the export pipeline in Trionic AI-Adalat.

The goal of this module is to convert AI-generated legal drafts into professional PDF and DOCX documents while maintaining:
- citation integrity
- secure storage
- printable formatting
- mandatory disclaimer banners

The export layer receives Markdown drafts from the agent pipeline, generates exports, uploads them to Supabase Storage, and returns signed download URLs.

This RFC mainly focuses on:
- Markdown → PDF/DOCX flow
- Puppeteer vs server-side PDF generation
- citation appendix rendering
- disclaimer injection
- secure storage strategy

---

# 2. Pipeline Placement

Classifier → Planner → Drafter → Citator → Reviewer → Translator? → [Export API] → Storage

| Stage | Responsibility |
|---|---|
| Drafter | Generates Markdown draft with citations |
| Citator | Validates citation integrity |
| Reviewer | Checks tone and completeness |
| Export API (this RFC) | Generates PDF/DOCX exports |

# 4. Markdown Rendering

The export layer receives Markdown drafts generated by the agent pipeline.

The rendering flow:
- parse markdown
- sanitize content
- convert markdown → HTML
- apply legal-document styling
- inject citations and disclaimer sections

Recommended libraries:
- `marked`
OR
- `markdown-it`

HTML templates will be used to maintain consistent formatting across exports.

---

# 5. PDF Generation Strategy

Two approaches were considered for PDF generation.

## Option A — Puppeteer (Recommended)

Flow:

```txt
Markdown → HTML → Chromium → PDF
```

Advantages:
- better HTML/CSS rendering
- easier legal-document formatting
- multilingual rendering support
- header/footer support
- cleaner citation layouts

Disadvantages:
- heavier runtime
- Chromium dependency

---

## Option B — Server-side PDF Libraries

Examples:
- PDFKit
- pdf-lib

Advantages:
- lightweight
- faster generation

Disadvantages:
- manual layout handling
- difficult styling for legal documents
- harder pagination management

---

# Decision

Puppeteer is recommended because legal documents require reliable formatting, print consistency, and better support for citations and disclaimers.

---

# 6. DOCX Export Strategy

DOCX exports will use the `docx` npm package.

The export layer will convert Markdown sections into editable Word document structures including:
- headings
- paragraphs
- lists
- citations
- disclaimer sections

DOCX support is important because exported drafts may still require manual editing before final use.

---

# 7. Citation Appendix Rendering

The project follows a strict “Citation-or-die” rule where every legal claim must map to a valid PageIndex node ID.

Exports should therefore include:
- inline citation references
- appendix section containing referenced acts and node IDs

Example:

```txt
References

[1] IPC Section 420
Node ID: IPC-1860/CH-XVII/S-420

[2] RTI Act Section 7
Node ID: RTI-2005/CH-II/S-7
```

Citation rendering flow:

```txt
Draft
   ↓
Extract Citations
   ↓
Deduplicate References
   ↓
Generate Appendix
   ↓
Append to Export
```

---

# 8. Disclaimer Banner

Every export must contain the disclaimer:

```txt
AI-generated draft — not legal advice.
```

Proposed placement:
- footer/header for PDFs
- top section for DOCX exports

The disclaimer should remain visible without affecting readability.

---

# 9. Storage Strategy

Generated exports will be stored in private Supabase Storage buckets.

Proposed bucket:

```txt
exports
```

Proposed path structure:

```txt
exports/users/{userId}/pdf/{documentId}.pdf

exports/users/{userId}/docx/{documentId}.docx
```

Example:

```txt
exports/users/user_42/pdf/legal-notice-2026.pdf
```

Exports should be accessed through signed URLs instead of public links.

---

# 10. Access Control & Security

The export system should:
- use private Supabase Storage buckets
- generate short-lived signed URLs
- enforce RLS on export metadata
- restrict users to their own exports
- avoid exposing PII in logs or URLs

Suggested signed URL expiry:
- 15 minutes
OR
- configurable through environment variables

---

# 11. Proposed API Endpoints

## Generate PDF

```http
POST /api/export/pdf
```

## Generate DOCX

```http
POST /api/export/docx
```

## Fetch Export Metadata

```http
GET /api/exports/:id
```

## Generate Download URL

```http
GET /api/exports/:id/download
```
